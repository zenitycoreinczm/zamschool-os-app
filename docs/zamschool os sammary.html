Architectural Charter

The 80/20 Reader vs Writer Axiom

ZamSchool OS functions on an asymmetric load distribution model. Over 80% of platform traffic is read-only. Parents, students, and advisors continuously consume timetable data, school announcements, attendance status checks, and report cards. Only 20% of traffic consists of active data mutations (teachers marking attendance, administrators adjusting classes, or bursars posting billing receipts).

Because of this massive imbalance, the platform utilizes strict write constraints while caching read structures. Write pipelines are protected by granular Postgres policies, whereas read endpoints are designed to bypass heavy calculations by leveraging localized cache structures, Redis buffers, and memory caches.

Bursty School Traffic Profiles
  • Morning Attendance Rush (07:30 - 08:30 AM): Massive concurrent bursts of teachers logging roll calls.
  • Term Result Releases (End of Term): Thousands of families fetching report cards simultaneously.
  • Bursty Announcements: Emergency updates sent via background queues to protect real-time threads.

The "Publish Once, Read Many" Philosophy

We separate draft academic records from public records. Teachers continuously key student grades in an unpublished draft state. Once complete and approved, a single command switches a boolean publish flag, triggering Supabase Realtime alerts that instantly update active parent and student portal feeds.

Notification & Realtime Caching Pipeline
flowchart TD
  A[Domain Write Event e.g. Mark Attendance] --> B[Record Inserted into Postgres Table]
  B --> C[Postgres Trigger inserts notifications table row]
  C --> D[Supabase Realtime Engine listens on WAL replication]
  D --> E[Broadcast WebSocket payload sent to authorized sessions]
  E --> F[Client UI receives event, increments unread indicator]
  B --> G[Bust Redis & In-Memory profile caches]

The 14 System Roles

ZamSchool OS is driven by 14 distinct roles mapped to physical school personas. Every account possesses a single `profiles.role` identifier that is evaluated by authorization middleware.

isAdminRole()

Evaluates if the actor handles core school resources. Returns true if the user matches one of the administrative variants:

role IN ('admin', 'principal', 'deputy_head', 'academic_admin', 'hr_admin', 'ict_admin', 'discipline_admin')
isSensitiveRole()

Limits super-user features (MFA adjustments, database preflights, custom domains) only to critical operations roles:

role IN ('super_admin', 'principal', 'bursar', 'ict_admin')
isFinancialRole()

Directs billing dashboard access. True only for staff roles with rights to fee catalogs and student payment entries:

role IN ('bursar', 'payments')

Setup & Onboarding Runbook

Based on the strict requirements of PILOT_SCHOOL_RUNBOOK.md, this step-by-step phased approach takes a new tenant school from system preflight to full go-live status. 

flowchart TD
  P0[Phase 0: Environment Preflight] --> P1[Phase 1: Generate Access Code]
  P1 --> P2[Phase 2: School Registration]
  P2 --> P3[Phase 3: Academic Setup]
  P3 --> P4[Phase 4: User Provisioning]
  P4 --> P5[Phase 5: Day-1 Smoke Tests]
  P5 --> P6[Phase 6: Live Sync & Audits]
Phase 0

Environment Preflight & Deployment

Verify platform readiness prior to introducing user flows. Configure the production environment variables, confirming no private secrets are present in public repositories, and trigger preflight diagnostics:

npm run pilot:preflight
Phase 1

Access Code Generation

Platform super-admins sign in at /app/super-admin to generate a single-use, 6-digit access code for the target tenant. This code is passed securely to the head teacher.

GET /api/super-admin/access-codes Outputs: { code: "918237" }
Phase 2

School Registration

The incoming head teacher opens /register, inputs the 6-digit access code, and completes the school profile. The registration action runs initializeSchoolDefaults, provisioning default departments and permissions inside the new tenant.

POST /api/auth/verify-access-code
POST /api/auth/register-school
Phase 3

Academic Structure

The school administrator designs the academic year, active school terms, streams, classes, and subjects using UI controls inside the Admin console:

Years & Terms

POST /api/admin/academic-years
POST /api/admin/terms

Classes & Streams

POST /api/admin/classes

Subjects & Lessons

POST /api/admin/subjects
POST /api/admin/timetable

Phase 4

User Provisioning (Teachers, Students & Parents)

Configure system roles in the required sequence to maintain relationship references:

  1. Create teacher records, mapping subject specializations and assigned streams (POST /api/admin/staff/invite).
  2. Enroll student profiles, binding them to stream classes.
  3. Register parents, linking them to students (POST /api/admin/relationships).
Phase 5

Day-1 Smoke Tests

Execute test operations with school coordinators to guarantee performance under real-world conditions:

  • Confirm Head Teacher first login triggers forced password change at /first-login.
  • Verify Teacher bootstrap dashboard takes under 2.5s at p95 latency.
  • Mark attendance for a single class and confirm instant WebSocket updates on parent views.
  • Upload an avatar to verify direct CDN loading.

Multi-Tenant Row-Level Security

Tenancy Isolation via school_id

Every table containing tenant-private rows is bound to a strict Postgres Row-Level Security (RLS) policy. Tenant boundaries are governed via a foreign key reference school_id. There are no global raw SQL statements accessible to users; the engine automatically screens out unauthorized records.

When a user authenticates, their database token contains metadata that points to their primary profile record. Postgres compares this profile data with its policy filters, bypassing the need for developers to manually inject WHERE school_id = X filters in routing logic.

Typical Policy (Academic Records) CREATE POLICY school_isolation_policy ON public.results AS RESTRICTIVE USING (school_id = get_my_school_id()) WITH CHECK (school_id = get_my_school_id());

Security Definer Functions

Because user metadata sits inside the private auth schema and user profiles reside in public tables, optimized SECURITY DEFINER functions are used to resolve actor context fast.

Function Security Level Behavior
get_my_school_id() AUTHENTICATED ONLY Resolves profile `school_id` from user credentials.
get_my_role() AUTHENTICATED ONLY Resolves actor role from cached database records.
is_sensitive_role() HIGH AUDIT Evaluates role risk for critical write mutations.
Hardening Actions (Migration 025)

By default, public database schemas grant execution rights on functions to public/anonymous roles. ZamSchool explicitly revokes helper execution privileges from anon and PUBLIC, limiting helper execution entirely to logged-in user tokens.

Cloudflare R2 File Storage

Pre-Authorized Upload Flow

To conserve server compute and memory, ZamSchool OS serves static and user uploads through Cloudflare R2. In production, files are not streamed through the main Next.js backend server.

When a user attaches files or updates an avatar, the client requests a pre-authorized signed upload URL via /api/files/authorize-upload. After passing security validation, the server returns a temporary target bucket URL. The client then streams the binary payload directly to R2.

Environment Variables (CDN_AND_R2.md)
R2_ENDPOINT = https://[account].r2.cloudflarestorage.com
R2_ASSETS_BUCKET = zamschool-assets
R2_PUBLIC_URL = https://pub-[hash].r2.dev

R2 Delivery Modes

The storage configuration supports two key delivery modes:

Mode Trigger Condition Output URL Shape
cdn Mode R2_PUBLIC_URL is set https://pub-[id].r2.dev/[school_id]/[key]
proxy Mode R2_PUBLIC_URL is blank /api/public/assets/[key]
Proxy Warning: If running in proxy mode, files are served through our backend server, consuming hosting bandwidth. Web apps and native applications must use cdn mode in production.

Caching & Latency Layers

To manage traffic spikes during exam results releases and morning registration, ZamSchool OS employs multiple layers of caching:

1. Client Memory Cache

Stable resources (such as active class stream metadata and school years) are stored inside local React/Native memory contexts.

TTL: 60 Seconds

2. Redis Role Caches

Stores authenticated security credentials, user permission role scopes, and request throttling limits.

Managed dynamically via ioredis

3. Platform API Budget

Employs a sliding window budget limiting client requests to prevent database performance degradation:

25 - 30 req/sec threshold
